# Issue #218 — Regression Repro Test Design Notes (2026-04-26)

## Bug recap
- Symptom: `IsHungAppWindow=true` on three concurrently-running WinUI3 sessions
  during long uptime. cdb stack on the UI thread was always:
  `Condition.wait` -> `BlockingQueue.push` -> `Surface.focusCallback`.
- Root cause (Surface.zig:3382 pre-fix):
  ```
  _ = self.renderer_thread.mailbox.push(.{ .focus = focused }, .{ .forever = {} });
  ```
  When the renderer mailbox is saturated (sustained renderer load, or a
  briefly stalled consumer thread), `.forever` parks the **UI thread** on
  `cond_not_full.wait` indefinitely. The UI message pump then stops draining
  Win32 messages, so the OS marks the window hung.
- Fix (already in flight on branch `fix-218-focus-mailbox-drop` by agent
  `a5b9a24de949fa1f1`): switch to `.instant` push and emit
  `log.warn("focus event dropped (renderer mailbox full)", ...)`.

## Test design
The bug is fundamentally a contract issue: `BlockingQueue.push(.forever)`
**will** block the caller indefinitely when the queue is full. That contract
is what makes the buggy `Surface.focusCallback` deadlock on the UI thread.
Conversely, `.instant` and `.ns` are bounded.

So the tests assert the contract, not the call site:
1. **`push forever blocks indefinitely when full`** — saturate a 2-slot
   queue, spawn a side thread to attempt a `.forever` push, sleep 100 ms
   on the test thread, assert via `std.atomic.Value(bool)` that the side
   thread **has not** completed. This is the mechanical proof of the
   bug shape. Cleanup pops one item and joins the thread under a 2 s
   bounded wait.
2. **`push with ns timeout returns 0 on full queue without hanging`** —
   same setup, `.{ .ns = 50 ms }`. Producer must return ≤ 400 ms with
   return value 0 and queue contents untouched. Proves the timeout
   contract a fixed callback can rely on.
3. **`focus saturation: forever hangs producer, instant drops cleanly
   (#218)`** — directly models the `focusCallback` path. Phase A uses a
   4-slot mailbox + 5 sequential `.forever` pushes on a side thread;
   asserts that exactly 4 pushes complete and the 5th is parked after
   200 ms. Phase B repeats with `.instant`; asserts all 5 complete with
   exactly 1 drop and total wall-clock under 200 ms.

### Self-bounded waiting (hard rule)
Every test sets explicit deadlines (`join_deadline_ns`, polling loops
with caps) so the test runner is **never** relied on as the timeout of
last resort. `zig build test --timeout` is "second line" per the brief.

### Memory ordering
Atomic flag publish uses release/acquire pairs and the producer publishes
the return value **before** flipping `done`, so a reader that sees
`done = true` always sees a coherent `push_return`.

## Running the tests

The file lives under `tests/` (outside the `src/` module path) so it
imports `blocking_queue.zig` as a named module rather than via a
relative path. Command:

```
zig test \
  --dep blocking_queue \
  -Mroot=tests/repro_focus_mailbox_hang.zig \
  -Mblocking_queue=src/datastruct/blocking_queue.zig
```

This sidesteps the project's main `build.zig` test wiring (which would
otherwise require touching `src/datastruct/main.zig` to add the import).

The queue itself is unchanged in the fix; only `Surface.zig` swapped its
timeout argument. So the tests intentionally pin the contract at the
data-structure level, where it is cheap to assert and impossible to
regress without also changing the queue API.

## Why we did NOT touch `src/Surface.zig`
- Owned by another agent in `fix-218-focus-mailbox-drop`; touching it
  here would force a merge later. The mechanical contract test is
  sufficient because the buggy code's signature on the UI thread comes
  entirely from the queue's `.forever` semantics.
- A higher-fidelity test that drives `focusCallback` directly would
  require constructing a real `Surface` — that needs an apprt, a
  renderer thread, a libxev loop, and Win32 surfaces. Not worth it for
  a contract whose deadlock is already mechanically reproduced in
  ≤ 200 ms by Test 3.
