### 1. このコードは何をしているか（目的と責務）
Ghostty 端末の **WinUI 3 バックエンドにおける描画・入力の実装** を担っています。
*   **描画のブリッジ:** Ghostty コア（`CoreSurface`）と WinUI 3 の `SwapChainPanel` を接続し、レンダラースレッドで生成された DirectX スワップチェーンを UI スレッドにバインドします。
*   **タブ管理:** `TabViewItem` 内のコンテンツとして動作し、タブタイトルの更新などを制御します。
*   **入力処理:** Win32 の `WM_KEYDOWN`/`WM_CHAR` を Ghostty 共通の `KeyEvent` に変換し、IME やサロゲートペアを考慮したステートマシンで管理します。
*   **OS 連携:** クリップボード操作、DPI スケール対応、マウスカーソル制御などの Windows 固有機能を提供します。

### 2. 依存関係
*   **Ghostty コア:** `../../Surface.zig` (コア状態), `../../App.zig`, `../../input.zig` (入力定義)。
*   **WinUI 3 / COM:** `winrt.zig`, `com.zig`, `native_interop.zig` を介して Microsoft.UI.Xaml や COM インターフェース（`ISwapChainPanelNative` 等）を操作。
*   **Windows API:** `os.zig` を介してクリップボード、メッセージング、DPI 取得などの Win32 関数を使用。
*   **サブコンポーネント:** `search_overlay.zig` (検索 UI), `key.zig` (キーマッピング)。

### 3. 設計パターン
*   **Proxy / Bridge パターン:** プラットフォーム抽象化レイヤー（`apprt`）の要求を、WinUI 3 固有の実装へ橋渡ししています。
*   **Deferred Initialization (遅延初期化):** `Loaded` イベントを待ち、コントロールが視覚ツリーに入ってから描画リソースをバインドするガードロジック。
*   **State Machine (状態管理):** `pending_keydown` により、非同期に発生するキーダウンと文字入力を正しくマージしています。
*   **COM インタラクション:** 手動参照カウント（`AddRef`/`Release`）によるリソース管理。

*   **COM 管理の安全性:** `release()` の呼び出しが手動であり、複雑な `errdefer` 網に頼っています。Zig の `defer` を活用したスマートポインタ的な抽象化があると、リソースリークや二重解放を防ぎやすくなります。
